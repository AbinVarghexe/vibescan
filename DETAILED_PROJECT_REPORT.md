## 1. Introduction

Modern software development increasingly relies on AI-powered coding assistants (GitHub Copilot, ChatGPT, Claude, etc.) to accelerate development. However, these tools occasionally "hallucinate" non-existent package dependencies, leading to build failures, supply chain vulnerabilities, and potential security exploits through typosquatting attacks. **VibeScan** is a comprehensive pre-publish security analysis tool that automatically detects AI-hallucinated dependencies, identifies typosquatting attempts, and provides actionable risk assessments before code reaches production.

The tool scans both NPM (`package.json`) and PyPI (`requirements.txt`) dependency files, validates packages against official registries, calculates multi-factor risk scores (0–100), and generates detailed reports in terminal, JSON, text, and PDF formats. VibeScan integrates seamlessly into CI/CD pipelines, preventing deployment of compromised dependencies.

---

## 2. Objectives

- **Validate Dependencies:** Cross-reference all declared packages against official NPM and PyPI registries in real-time.
- **Detect Hallucinations:** Identify non-existent packages generated by AI coding assistants that could be exploited via namespace squatting.
- **Prevent Typosquatting:** Use Levenshtein distance algorithms to detect malicious packages impersonating legitimate libraries (e.g., `requets` vs. `requests`).
- **Risk Quantification:** Calculate weighted risk scores (0–100) based on package age, download popularity, registry status, and similarity metrics.
- **Multi-Format Reporting:** Generate terminal (color-coded), JSON (machine-readable), text (human-readable), and PDF (professional) reports.
- **CI/CD Integration:** Exit with non-zero codes on critical findings to halt unsafe builds automatically.

## 4. System Architecture

VibeScan follows a **modular pipeline architecture** with clear separation of concerns:

```
CLI Entry Point (cli.py)
      │
      ▼
Project Scanner
  ├── parse_package_json()      (parsers.py)
  ├── parse_requirements_txt()  (parsers.py)
      │
      ▼
Registry Validation Layer
  ├── check_npm_registry()      (registry_checker.py)
  ├── check_pypi_registry()     (registry_checker.py)
  └── fetch_package_metadata()
      │
      ▼
Typosquatting Detection
  ├── levenshtein_distance()    (typosquat_checker.py)
  ├── compare_with_popular()    (typosquat_checker.py)
  └── similarity_threshold()
      │
      ▼
Risk Scoring Engine
  └── calculate_risk()          (scorer.py)
      │
      ▼
Report Generation
  ├── Terminal Output           (reporter.py)
  ├── JSON Export               (reporter.py)
  ├── Text Report               (reporter.py)
  └── PDF Report                (pdf_report.py)
```

**Data Flow:**
1. User invokes `vibescan [path]` command
2. CLI discovers `package.json` and/or `requirements.txt` files
3. Parsers extract package names and version constraints
4. Registry checkers validate each package against NPM/PyPI APIs
5. Typosquatting checker compares against 500+ popular packages
6. Risk scorer calculates 0–100 risk based on weighted factors
7. Reporter categorizes results (Safe / Suspicious / Critical) and outputs

## 5. Key Features Implemented

### 5.1 Automatic Dependency Discovery
- Recursively scans project directories for `package.json` (NPM) and `requirements.txt` (PyPI).
- Supports monorepos and nested project structures.
- Handles mixed-language projects (Node.js + Python).

### 5.2 Multi-Registry Validation
**NPM Registry:**
- `https://registry.npmjs.org/<package>` — validates existence, retrieves metadata.
- Extracts creation date, latest version, download counts (via npm-stat API).

**PyPI Registry:**
- `https://pypi.org/pypi/<package>/json` — validates package existence.
- Retrieves release dates, download statistics, maintainer info.

### 5.3 Hallucination Detection
Identifies packages that **do not exist** in official registries:
```
❌ react-native-vibe-sensor  → Not found in NPM
❌ sklearn-advanced-models   → Not found in PyPI
```
These are flagged as **CRITICAL** (risk score 95–100) as they indicate:
- AI code generation errors
- Potential namespace squatting opportunities for attackers

### 5.4 Typosquatting Protection
Uses **Levenshtein distance** to detect malicious package names similar to popular libraries:

| Malicious Package | Legitimate Package | Distance | Risk |
|---|---|---|---|
| `requets` | `requests` | 1 | CRITICAL |
| `python-requet` | `requests` | 8 | SUSPICIOUS |
| `reacts` | `react` | 1 | CRITICAL |
| `expres` | `express` | 1 | CRITICAL |

**Popular Package Database:**
- NPM: 250+ packages (react, lodash, express, axios, etc.)
- PyPI: 250+ packages (requests, numpy, pandas, flask, etc.)

### 5.5 Risk Scoring Algorithm

Each package receives a **0–100 risk score** calculated as:

```python
risk_score = (
    package_not_found        × 95.0  +  # Critical
    typosquat_similarity     × 40.0  +  # High
    low_download_count       × 20.0  +  # Medium
    recent_package_age       × 15.0     # Low
) / total_weight
```

**Risk Categories:**
- **Safe (0–39):** Established, popular packages with no issues
- **Suspicious (40–74):** Low downloads, new packages, or moderate similarity
- **Critical (75–100):** Non-existent or obvious typosquatting
